## 1. Executive Summary

The system is a production Next.js 15 full-stack web application delivering real-time FIFA World Cup 2026 intelligence. It aggregates live match data from api-football.com, overlays prediction-market probability signals from Kalshi and Polymarket, generates AI commentary via Claude Haiku, and hosts a fully operational team-anthem music platform with social-sharing and listen-tracking.

**Deployment #1 is live.** The platform is receiving real traffic. The data model, API surface, and core features are stable. The codebase is organized for a single sport / single tournament, but it was deliberately architected to avoid hard-coupling: vendor adapters are isolated, the metric store is key/value, the auth and tenancy layers are sport-agnostic, and a `v2/` API namespace already exists.

**Key numbers (as built):**
- 14 Prisma models
- ~30 API routes
- 10 page routes
- 6 core data sources (3 external APIs, 1 LLM, 1 prediction-market pair, 1 flight-tracking network)
- 4 proprietary named metrics (Momentum Pulseâ„˘, Strike Clockâ„˘, Score Volatilityâ„˘, Clutch Indexâ„˘)
- 3 AI commentary personas
- 1 white-label partner model (schema only; no embed widget shipped)
- 1 sponsorship + ad-slot framework (schema + admin; no self-serve UI)

**Strategic position:** This deployment is further along than its name suggests. The intelligence engine, the tenancy model, and the versioned API are the strongest SIE-native assets in the codebase. The four highest-risk legacy items â€” 2-participant match model, vendor-keyed fixture IDs, football-only outcome math, and goal-centric metrics â€” are localised and can be unwound post-launch without touching live infrastructure.

**Key architectural pattern:** server components fetch data at request time (with `next: { revalidate: N }` caching); client components hydrate and maintain local interaction state. The audio system (`AudioContext`) is the only globally shared runtime state.

**Shared audio state (`AudioContext.tsx`):**
- React context wrapping `new Audio()` in a client-side `useEffect` (one audio element at root).
- Exposed via `useAudio()` hook.
- `AudioProvider` wraps root layout; `FloatingMiniPlayer` sits at root level.
- Features: play, pause, seek, volume, mute, loop (none/all/one), shuffle.
- Playlist-aware: `play(track, playlist, index)` sets context + starts playback.
- Listen tracking: fires POST at 10-second mark; flushes on pause/track-change.

## 4. Data Sources

### 4.1 api-football.com (Primary Match Data)

- **Plan:** Pro (league=1 FIFA World Cup, season=2026)
- **Base URL:** `https://v3.football.api-sports.io`
- **Auth:** `x-apisports-key` header (`API_FOOTBALL_KEY` env var)
- **Used for:** Live scores, match status, elapsed time, goal events, lineups, player events
- **Routes:** `/fixtures/events?fixture=N`, `/fixtures/lineups?fixture=N`, `/fixtures?league=1&season=2026`
- **Caching:** 10s revalidation for live events; 2 min in-memory for today's matches; 1 hr for past
- **Fallback:** Deterministic seeded simulation (`lib/simulation.ts`) when key absent or API fails
- **Isolation:** `lib/footballData.ts`

### 4.2 Kalshi (Match Outcome Markets)

- **Access:** Public API (no key required)
- **Used for:** Home win / draw / away win market prices per match
- **Ticker strategy:** Builds candidate contract tickers from team names + date Â±1 day + both team orderings; falls back to search by KXFIFAGAME series
- **Caching:** 10 seconds per match pair (in-memory)
- **Returns:** `{ home_win, draw, away_win, volume, tickers, source }`
- **Isolation:** `lib/kalshi.ts`

### 4.3 Polymarket (Tournament & Group Odds)

- **Access:** Gamma API (public, no key required)
- **Used for:** Tournament winner probabilities (all teams), group winner odds (per group)
- **Caching:** Tournament markets: 5 min; group markets: 10s
- **Mapping:** `NAME_TO_TLA` lookup maps Polymarket team name strings â†’ 3-letter codes
- **Isolation:** `lib/polymarket.ts`

### 4.4 Anthropic (Claude Haiku 4.5)

- **Model:** `claude-haiku-4-5-20251001`
- **Auth:** `ANTHROPIC_API_KEY` env var
- **SDK:** `@anthropic-ai/sdk 0.104.2`
- **Used for:** Match commentary (`/api/matches/[id]/commentary`)
- **Parameters:** `max_tokens: 512`, 3 system personas
- **No streaming:** Response is buffered and returned as JSON array of lines
- **Isolation:** Inline in commentary route handler

### 4.5 OpenSky Network (Flight Tracking)

- **Access:** Public REST API (no key required)
- **Used for:** Live aircraft positions between team basecamps and stadiums
- **Routes:** `/api/flight-density`, `/api/flight-paths`

## 5. Metrics & Intelligence Engine

The Match DNAâ„˘ system is the core proprietary IP. It translates a `GoalEvent[]` array into four named visual analytics primitives. All four are rendered on the match detail page and on the home dashboard for the featured match.

### 5.1 GoalEvent Schema

```ts
interface GoalEvent {
  minute: number;    // 1â€“120
  team: string;      // team name
  scorer: string;    // player name
  assist: string | null;
  isOwnGoal: boolean;
  isPenalty: boolean;
}
```

Goal events are sourced from:
1. **api-football.com live events** (primary â€” real data)
2. **Deterministic simulation** (`simulateGoals` in goals route â€” fallback when API returns empty despite score > 0)

The simulation generates plausible goal events keyed by `fixture * 7 + 13` seed, using real squad player names from the Prisma `Player` table.

### 5.2 Momentum Pulseâ„˘

**What it measures:** Score advantage at each 15-minute checkpoint.

**Implementation:**
- Segments the match into 15-min windows: 15â€˛, 30â€˛, 45â€˛, 60â€˛, 75â€˛, 90â€˛.
- At each checkpoint, counts net goals scored by each team to date.
- Renders a horizontal 0â†’90â€˛ timeline bar with coloured goal-dot markers at exact minute positions.
- Colour coding: home lead (blue), away lead (red), level (amber).
- Live pulse indicator shown when match is in-progress.
- **Scaffold:** renders even at 0â€“0 with empty bar and "awaiting" state.

### 5.3 Strike Clockâ„˘

**What it measures:** Goal timing rhythm and scoring cadence.

**Implementation:**
- `first`: minute of the first goal.
- `avgGap`: average minutes between consecutive goals.
- `rhythm` label derived from avgGap:
  - `avgGap < 10` â†’ "Explosive"
  - `avgGap < 20` â†’ "High-Scoring"
  - Goals clustered after 70â€˛ â†’ "Late Drama"
  - Otherwise â†’ "Steady"
- **Scaffold:** shows `"â€”"` / `"Awaiting"` when no goals.

### 5.4 Score Volatilityâ„˘

**What it measures:** Match drama through lead changes.

**Implementation:**
- Counts lead changes (home leading â†’ away leading or vice versa, across goal sequence).
- Counts equalisers (goals that restored parity).
- `svLabel` derived:
  - `â‰Ą 3 lead changes` â†’ "High Drama"
  - `1â€“2 lead changes` â†’ "Tense"
  - `> 0 equalisers` â†’ "Shifted"
  - Otherwise â†’ "Dominant"
- **Scaffold:** `svLabel = "Level"`, emoji `"âš–ď¸Ź"` when no goals.

### 5.5 Clutch Indexâ„˘

**What it measures:** Per-scorer weighted impact; rewards decisive late goals.

**Implementation:**
- Per goal, compute base weight:
  - Lead-changing goal â†’ `3.0`
  - Equalising goal â†’ `2.5`
  - Score-extending goal â†’ `1.0`
  - Own goal â†’ `0.5`
- Multiply by `1.5` if `minute â‰Ą 80` (late-game multiplier).
- Aggregate per scorer; rank descending.
- Displays top scorers with their weighted Clutch Index score.
- **Scaffold:** `"Awaiting first goalâ€¦"` placeholder when `ci.length === 0`.

### 5.6 Derived Probability Metrics

Located in `lib/probabilities.ts`:

- **`liveWinProbability()`** â€” Blends:
  1. Pre-match win probability from Polymarket tournament odds (via `preMatchWinProbFromTournamentOdds`).
  2. Score-based logistic model (goal difference Ă— time-progress factor).
  - Draw probability: peaks at ~28% pre-match, decays with score separation.
  - Output: `{ home, draw, away }` probabilities summing to 1.

- **`harvilleAdvanceProbability()`** â€” Harville's formula: probability a team advances from group stage given all group teams' tournament win probabilities.

- **`tournamentPathProbabilities()`** â€” Given P(win tournament) + P(advance group), derive P(reach R16/QF/SF/Final) via geometric progression.

- **`preMatchWinProbFromTournamentOdds()`** â€” P(home wins) = `home_odds / (home_odds + away_odds) + 0.03` (3% home advantage).

### 5.7 Tournament Entropy (Implicit)

The `GroupWinnerTickers` component visualises effective group entropy via Polymarket group winner probability bars. No explicit "entropy" value is computed; the visual implies it.

### 5.8 Simulation Engine (`lib/simulation.ts`)

Fallback deterministic match state generator:

- `simulateStats(fixture, elapsed, teamCode)` â†’ per-team `LiveMetric` values (possession, shots, corners, fouls, cards).
- `simulateMarkets(fixture)` â†’ seeded home/draw/away prices (`0.51 Â± 0.06` envelope).
- `elapsedFromDate(date)` â†’ match minute from wall-clock time (accounts for 45-min halves + 17-min half-time).
- `statusFromElapsed(elapsed)` â†’ `NS|LIVE|HT|FT`.
- All outputs are deterministic for a given `fixture` int â€” same input always produces same output.

## 10. Immediate Migration Recommendations

These are low-risk, zero-migration actions that can begin immediately without touching the live deployment. They accrue no new debt.

### 10.1 Naming Convention Freeze (Do Now)

All new code (tables, routes, components, type names) must use SIE vocabulary:
- `Event` not `Match` (for new additions)
- `Athlete` not `Player`
- `Competitor` not `Team` (for new multi-entrant contexts)
- `/api/events/` not `/api/matches/` (for new route additions)
- `components/event/` not `components/match/` (for new components)
