## Strategic Decisions

**1. World Cup 2026 is Deployment #1, not the product.**
The platform is formally repositioned as the Sports Intelligence Engine™ (SIE). The SIE is the product. World Cup 2026 is the first implementation. All future architecture must prioritise platform reusability.

**2. SIE target hierarchy is the governing vocabulary.**
`Sport → Competition → Event → Venue → Team → Athlete → Hospitality → Travel → Sponsor → Metric`
All new code, routes, models, and components must use this vocabulary. Deviation is technical debt.

**3. The live deployment is not reset.**
World Cup 2026 stays live exactly as deployed. All generalization and migration work is deferred to the post-tournament window. The documentation baseline created today governs what happens then.

**4. Five protected architectural assets designated.**
Per the SIE Constitution: `LiveMetric`, `Match DNA™`, `WhiteLabelPartner`, `AudioStream`, and the `Vendor Adapter Pattern` are Protected Architectural Assets. They may evolve. They must not be casually replaced.

**5. Four migrations sequenced for post-launch.**
In priority order: `Match → Event`, `Player → Athlete`, `GoalEvent → MomentEvent`, `Match Simulator → Event Simulator`. These are the gates to multi-sport capability.

**6. Naming convention freeze begins immediately.**
All new code created from today uses SIE vocabulary. This costs nothing and prevents debt from compounding between now and the post-launch migration.

**7. `Match.fixture` (vendor integer as primary key) is the highest-risk debt.**
Acknowledged as requiring the first post-launch schema migration. Every multi-sport expansion depends on resolving this before a second sport data source is integrated.

**8. Sponsorship activation is the fastest path to revenue.**
The schema is complete. Activation requires display components and CRUD routes, not new architecture. Prioritised as near-term commercial action.

**9. White-label embed widget is the highest near-term commercial priority.**
`WhiteLabelPartner` model, `embedKey`, and role system all exist. The embed widget, partner portal, and branding injection are the missing pieces. Estimated 2–3 sprints to a shippable v1.

**10. Audio is a first-class platform capability, not a sports feature.**
The `AudioStream` model and `AudioContext` system are designated as the universal media layer. Future uses: AI commentary audio, race radio, match recaps, daily briefings. All audio assets should flow through this model.

## Architecture Progress

**Discoveries:**

- `LiveMetric` is the cleanest SIE primitive already in production. A generic key/value/time store with string-typed metric names. Zero sport coupling. The entire SIE Metric Registry described in the Constitution can be stored here without schema changes.

- The `WhiteLabelPartner` model is more complete than previously understood. `embedKey`, `domain`, `primaryColor`, `planName`, `active` — all fields present. The commercial gap is in the runtime layer, not the data layer.

- The audio system (`AudioStream` + `AudioContext` + Vercel Blob + admin upload tooling) is the most architecturally complete feature on the platform. It is a reference implementation for how the media layer should work.

- The sponsorship schema is effectively inactive. `Sponsor` and `AdSlot` models are production-ready with impression/click counters. Nothing renders in the product. Revenue-ready infrastructure sitting idle.

- `Match.fixture` (api-football vendor integer) being a unique constraint and a FK in `Prediction.fixtureId` is a foundational architectural constraint. This was identified as the highest-risk item for multi-sport expansion.

**Conclusions:**

- The platform is architecturally further along than its World Cup branding suggests. The tenancy, API versioning, metric store, and vendor isolation are SIE-grade already.
- The product layer (pages, components, metrics) is tightly football-specific. The infrastructure layer is not.
- The `GoalEvent → MomentEvent` migration is the single highest-value product-layer change. It makes all four named metrics cross-sport without changing their visual design.
- An internal opaque event ID must be introduced before any second sport data source is integrated. This is a prerequisite, not an option.

**White-label readiness overall score: 5/10.** The hardest parts of multi-tenancy are built. The customer-facing parts are not.

**After today:**

The five documents created today are the bridge between what was built and what the platform becomes. They do not change any code. They change the frame within which all future code is written.

## Risks Identified

| Risk | Severity | Notes |
|---|---|---|
| `Match.fixture` vendor ID as primary key | Critical | Every multi-sport expansion blocked until resolved. First post-launch migration. |
| 2-participant event model (homeTeam/awayTeam) | High | Pervasive across schema, components, metrics, and simulation. Widest-surface migration. |
| Goal-only metric engine | High | Brand promise of cross-sport intelligence cannot be delivered until `MomentEvent` replaces `GoalEvent`. |
| Football-only outcome math (home/draw/away) | High | Win probability system unusable for any sport without a draw. |
| Goals simulation fallback may have been lost in merge | Medium | The `simulateGoals` fallback in the goals API route may have been overwritten during a prior merge conflict resolution. Needs verification. |
| AI commentary not cached | Medium | Regenerated on every request. At match-day scale this is a significant Anthropic API cost. |
| In-memory cache not shared across serverless instances | Medium | Adequate at current scale; becomes a correctness issue at multi-instance deployment. |
| Sponsorship schema inactive | Medium | Revenue-ready infrastructure with $0 live. Match-day inventory is being lost every day. |
| White-label commercially unbuilt | Medium | B2B revenue channel exists at schema level only. |
| Naming drift in new code | Low | Without an enforced freeze, new code creates new debt. Mitigated by convention; not enforced by tooling. |
